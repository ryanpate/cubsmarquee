# Mass-Production SD Card Image — Design

**Date:** 2026-04-10
**Status:** Approved for planning

## Goal

Produce a flashable Raspberry Pi SD card image so new Cubs Marquee units can be mass-produced. A buyer flashes the image, powers the Pi on, configures their WiFi and API keys via a built-in web admin panel, and the scoreboard runs. No per-unit code changes, no SSH access, no terminal use required by the buyer.

## Approach

**Golden master clone with first-boot reset.** Set up one pristine Pi by hand, run a stripping script that removes all per-unit state and installs a first-boot service, shut it down, and capture its SD card to a `.img` file on a Mac. Each new Pi flashed from that image regenerates its identity and walks the buyer through setup on first boot.

This was chosen over a chrooted build script because the `rgbmatrix` library is painful to install outside a real Pi, and over a network-based provisioner because units must be configurable by non-technical buyers without a network already in place.

## Buyer Experience

1. Flash `cubsmarquee-vYYYY-MM-DD.img` to an SD card with Raspberry Pi Imager.
2. Insert SD card into Pi, connect LED matrix, power on.
3. Wait ~60 seconds. The LED matrix shows scrolling instructions:
   ```
   SETUP REQUIRED → Connect to WiFi: CubsMarquee-Setup → Open: cubsmarquee.local/admin
   ```
4. Buyer connects their phone to the `CubsMarquee-Setup` WiFi network broadcast by the Pi.
5. Buyer opens `cubsmarquee.local/admin` in their phone browser, enters their home WiFi credentials and (optionally) API keys.
6. Pi reboots onto the home WiFi. The setup display exits and the normal scoreboard runs.

## Architecture

Three deliverables and two code changes.

### Deliverables

| File | Runs on | Purpose |
|------|---------|---------|
| `prepare_for_image.sh` | Golden master Pi | Strips per-unit state, installs first-boot hook, shuts down |
| `capture_image.sh` | Mac (or Linux) | Reads the SD card to a `.img` file via `dd` |
| `first_boot.sh` + `first-boot.service` | Each new Pi, once | Regenerates identity, expands filesystem, then disables itself |

### Code changes

| File | Change |
|------|--------|
| `setup_display.py` (new) | Display handler that scrolls setup instructions on the matrix until WiFi + config are present |
| `main.py` | At startup, check for setup state and run `SetupDisplay` until configured before entering normal cycle |

## Component Details

### `prepare_for_image.sh`

Run as root on the working golden master Pi as the very last step before shutdown. Idempotent where possible.

**Stop services and clear runtime state**
- `systemctl stop cubs-scoreboard wifi-manager wifi-web-config`
- Truncate `/home/pi/scoreboard_logs/*` and `/var/log/cubs-scoreboard/*`
- `journalctl --rotate && journalctl --vacuum-time=1s`

**Strip per-unit identity**
- Delete `/etc/ssh/ssh_host_*` (regenerated by `first-boot.service`)
- Delete `/home/pi/config.json`
- Delete `/etc/NetworkManager/system-connections/*`
- Delete `/etc/wpa_supplicant/wpa_supplicant.conf` if present
- Reset hostname to `cubsmarquee` in `/etc/hostname` and `/etc/hosts`
- Clear `~/.bash_history`, `/root/.bash_history`
- Clear `~/.ssh/`, `/root/.ssh/`
- Clear `~/.git-credentials`, `~/.config/git/`

**Clear caches**
- `apt-get clean`
- Remove `/var/cache/apt/archives/*.deb`
- Clear `/tmp/*`, `/var/tmp/*`

**Preserve (do not touch)**
- `cubs_facts.json`, `pga_facts.json`, `bible_facts.json`, `bible_verses.json`
- `logos/`, `logos_2/`, `fonts/`
- All `.png` weather and brand assets
- All Python source files in `/home/pi/cubsmarquee/`

**Install first-boot hook**
- Copy `first_boot.sh` to `/usr/local/sbin/first_boot.sh` (mode 0755)
- Copy `first-boot.service` to `/etc/systemd/system/first-boot.service`
- `systemctl enable first-boot.service`

**Write image version stamp**
- `echo "vYYYY-MM-DD" > /etc/cubsmarquee-version`

**Final**
- `sync && shutdown -h now`

### `first-boot.service` and `first_boot.sh`

Oneshot systemd unit, ordered before `cubs-scoreboard.service` and `wifi-manager.service`. Runs once on the first boot after flashing, then disables itself.

`first_boot.sh` does:
1. Regenerate SSH host keys: `ssh-keygen -A`
2. Expand the root filesystem to fill the actual SD card: `raspi-config nonint do_expand_rootfs`
3. Touch `/home/pi/.first-boot-complete`
4. `systemctl disable first-boot.service`
5. `reboot`

The reboot lets the expanded filesystem and regenerated keys take effect cleanly before any normal service starts.

WiFi AP mode does not need to be started here — the existing `wifi-manager.service` already starts AP mode automatically when no known networks are configured, which will be true on a fresh image.

### `setup_display.py`

A new display handler following the same pattern as `weather_display.py` and other off-season handlers.

**Triggered when:** `/home/pi/config.json` is missing OR `iwgetid -r` returns empty (not connected to WiFi).

**Behavior:**
- Renders a header stripe in `Colors.CUBS_BLUE` with the title `SETUP`
- Scrolls the instruction text in yellow:
  ```
  Connect phone to WiFi: CubsMarquee-Setup    Open: cubsmarquee.local/admin
  ```
- Polls every 10 seconds for both conditions to clear
- Returns control to `main.py` once both conditions are satisfied (no reboot required for the display to switch — `main.py` falls through to the normal cycle)

**Reuses:** existing `ScoreboardManager`, `Colors`, `Fonts`, and the `DisplayHandler` base class from `scoreboard_config.py`.

### `main.py` change

At startup, before entering the normal game-cycle loop:

```python
def needs_setup() -> bool:
    if not os.path.exists("/home/pi/config.json"):
        return True
    try:
        result = subprocess.run(["iwgetid", "-r"], capture_output=True, text=True)
        return not result.stdout.strip()
    except Exception:
        return True

if needs_setup():
    SetupDisplay(scoreboard).run_until_configured()
# fall through to normal cycle
```

`SetupDisplay.run_until_configured()` blocks (with shutdown-signal awareness) until both conditions clear.

### `capture_image.sh`

Runs on the user's Mac after the golden Pi is shut down and its SD card is in a USB reader.

1. Run `diskutil list` and prompt the user to type the device identifier (e.g., `disk4`). Never auto-pick.
2. Confirm the choice by re-displaying `diskutil info` for that disk and requiring a typed `yes`.
3. `diskutil unmountDisk /dev/diskN`
4. `sudo dd if=/dev/rdiskN of=cubsmarquee-vYYYY-MM-DD.img bs=4m status=progress`
5. Print the resulting file size and `sha256` checksum.

No shrinking step. The image will be the full size of the source SD card. Use the smallest SD card that fits everything as the golden master (8 GB or 16 GB) to keep file size reasonable. Pi Imager compresses on flash, so flash time is not impacted.

## Versioning

- Image filename: `cubsmarquee-v{YYYY-MM-DD}.img`
- `/etc/cubsmarquee-version` written by `prepare_for_image.sh` so any flashed unit can report its source image
- A `docs/IMAGING.md` written for the project owner with the full workflow, golden-master update steps, buyer-facing setup card text, and troubleshooting

## Testing

Manual end-to-end test required on real hardware:

1. Run `prepare_for_image.sh` on the golden master.
2. Capture with `capture_image.sh` to a `.img` file.
3. Flash the `.img` to a fresh SD card.
4. Boot a second Pi from the new card.
5. Verify the setup display appears on the matrix within ~90 seconds.
6. Verify `CubsMarquee-Setup` AP appears and `cubsmarquee.local/admin` is reachable.
7. Submit WiFi credentials and confirm the Pi reconnects on the home network.
8. Verify the normal scoreboard starts after configuration.
9. Verify SSH host key fingerprint differs from the golden master.
10. Verify `/etc/cubsmarquee-version` is present and correct.

## Out of Scope

- Automated build pipeline or CI image production
- Image shrinking with `pishrink`
- OTA updates (`marquee-update.timer` already handles this independently)
- Production of a printed setup card (text content lives in `docs/IMAGING.md`; physical printing is the project owner's responsibility)
- Multi-architecture or alternative-board support
